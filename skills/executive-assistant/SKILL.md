## Task Management

### Daily Task Review
Start each day with:
1. **Review commitments** - What's due today?
2. **Check calendar** - What meetings/events?
3. **Prioritize** - What's most important?
4. **Time block** - When will each task happen?
5. **Identify blockers** - What might get in the way?

### Task Capture Template
When capturing new tasks:
```
Task: [Clear, actionable description]
Project: [Which project/client]
Due: [Date/time or "flexible"]
Priority: [High/Medium/Low]
Time estimate: [How long]
Dependencies: [What needs to happen first]
Next action: [Very next physical step]
```

### Priority Framework
**P1 - Must do today:**
- Client commitments with deadlines
- Time-sensitive opportunities
- Blockers for others

**P2 - Should do this week:**
- Important but not urgent
- Proactive work
- Relationship building

**P3 - Do when possible:**
- Nice to have
- Long-term improvements
- Learning/exploration

### Weekly Review Checklist
Every week:
- [ ] Process all inboxes to zero
- [ ] Review calendar for next 2 weeks
- [ ] Update project statuses
- [ ] Identify stalled items
- [ ] Plan next week's priorities
- [ ] Review goals and progress
- [ ] Clear completed tasks
- [ ] Capture any loose threads

## Meeting Management

### Pre-Meeting Prep
Before any meeting:
1. **Purpose** - Why are we meeting?
2. **Desired outcome** - What should be true after?
3. **Agenda** - What topics to cover?
4. **Materials** - What do I need to bring/share?
5. **Questions** - What do I need to ask?

### Converting Notes to Actions
After every meeting:
1. **Extract action items** - What did I commit to?
2. **Assign due dates** - When does each need to happen?
3. **Add to task system** - Don't rely on notes alone
4. **Send summary** - Share notes with attendees if appropriate
5. **Schedule follow-ups** - Block time for commitments

## Email Management

### Email Processing Rules
- **2-minute rule:** If it takes <2 min, do it now
- **Defer:** If it needs more time, schedule when
- **Delegate:** If someone else should handle it
- **Delete/Archive:** If no action needed

### Email Response Priorities
**Respond within 4 hours:**
- Client requests
- Time-sensitive opportunities
- Blocking someone else

**Respond within 24 hours:**
- Most professional correspondence
- Non-urgent questions
- Scheduling requests

**Respond within week:**
- FYI/informational
- Low-priority requests
- Newsletters to actually read

## Staying on Track

### Focus Techniques

**Time Blocking:**
- Block 2-4 hour chunks for deep work
- Batch similar tasks together
- Protect focus time from meetings
- Build in buffer time

**Pomodoro Variation:**
- 50 minutes focused work
- 10 minute break
- After 4 cycles, longer break

**Daily Shutdown:**
1. Review what got done
2. Update task list
3. Identify tomorrow's top 3
4. Clear workspace
5. Set out-of-office or close apps

### Procrastination Busters
When stuck:
- **Smallest next action** - What's the tiniest step?
- **2-minute start** - Commit to just 2 minutes
- **Change environment** - Move locations
- **Body double** - Work alongside someone
- **Reward scheduling** - Treat after completion

### Energy Management
Match task type to energy:
- **High energy:** Creative work, strategic thinking, difficult conversations
- **Medium energy:** Meetings, email, collaboration
- **Low energy:** Admin, filing, routine tasks

### Taskwarrior

Derek uses Taskwarrior for task management. **Always use Taskwarrior** for creating, tracking, and completing tasks.

**Common Commands:**
```bash
# Create a task
task add "Task description" priority:H project:soapbox due:tomorrow

# List pending tasks
task list

# Complete a task
task <id> done

# View task details
task <id> info

# Modify a task
task <id> modify priority:M due:friday

# List tasks by project
task project:soapbox list

# Show today's tasks
task due:today list
```

**Priority Levels:**
- `priority:H` - High (urgent/blocking)
- `priority:M` - Medium (important)
- `priority:L` - Low (can wait)

**GitLab-Linked Tasks:**
Tasks synced from GitLab have custom attributes:
- `gitlab_id` - Unique GitLab identifier
- `gitlab_type` - "issue" or "mr"
- `gitlab_url` - Link to GitLab item
- `gitlab_project` - Project name

```bash
# List GitLab tasks
task +gitlab list

# View GitLab-specific info
task gitlab_project:soapbox list
```

**When Derek asks you to:**
- "Add a task" ‚Üí Use `task add`

- "What's on my plate?" ‚Üí Use `task list` or read the daily tasks file

### Taskwarrior & Document Sync

**IMPORTANT:** When modifying any document that contains tasks (schedules, task files, meeting notes, etc.), you MUST keep Taskwarrior in sync.

**When adding tasks to a document:**
1. First check if the task exists in Taskwarrior: `task project:<project> list` or search
2. If not found, add it to Taskwarrior: `task add "Task description" project:<project> priority:H due:<date>`
3. Then add it to the document

**When marking tasks complete in a document:**
1. Mark it complete in the document (change `- [ ]` to `- [x]`)
2. Find the corresponding task in Taskwarrior: `task project:<project> list`
3. Mark it done in Taskwarrior: `task <id> done`

**When building schedules or daily task files:**
1. If user requests adding a new task, add to BOTH the document AND Taskwarrior
2. If user says they completed a task, update BOTH the document AND Taskwarrior
3. Always verify sync by checking both systems

**Workflow Example:**
```bash
# User says: "Add a task to book Will's hotel"
task add "Book Moxy hotel room for Will" project:nostrnights priority:H due:2026-01-20
# Then add to the schedule/task document

# User says: "I booked Will's hotel"
task project:nostrnights list  # Find the task ID
task <id> done                  # Mark complete in Taskwarrior
# Then mark [x] in the document
```

This ensures the daily task files (auto-generated from Taskwarrior) stay accurate and nothing falls through the cracks.
